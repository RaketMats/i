# The learning loop
This is the main engine for work:
- A place to collect needs, to visualise problems. It is similar to Problem Space but here in depth studies are done, in the form of e.g. user journeys, interviews.
- A place to involve people of different roles.
  - The main driver here is 'customers dont know what they need until using the product'(Brooks)
- The real. A place to test solutions to the needs found. This can be the product but more likely mockups, prototypes in various form, real enough for learning to be done.
- A simple way to build applications. Examples are to use abstract base classes that corresponds and nicely solves objekt hierarchys on business and enterprise level
  - A design pattern is [ABCDesEntOrgSociety pattern](the_abcdesentorgsoc.md)  

# Then
- Reflection over planning. Dont plan too much. Do and write in the design.log what has been done and how it went. 
- The schoolbook approach to discovery and learning is to plan experiments carefully, deciding on a problem, a hypothesis and how to test it. In practice this is very hard to do. Instead, just do but carefully write down what has been done and how it went. 
- The key is to continously return to all the material, the design.log shall keep problem space brief, just an overview and then work is done in separate, preferralby small, easy to sort and search documents and models. 
- Dont analyse, structre, try to order things nicely - because it very seldom adds value. More often than not is is waste, never needed. Instead, write down carefully. Return to and re-read. Important things will rise to the surface.
- Use all the 'helper material':
  - Architectural frameworks such as Edgy that helps give structure and meaning.
  - Reference business architecture patterns such as ABCDesEnt since it will be built this way anyway and since it involves all necessary parts including end users and the cost/income reconcilitation and use case vs functionality
  - Principles such as clear boundaries, grids, 'Gestalt', that help the mind sort out and stay sane.
  - All the Analog Helpers and Apophtegmatas since there is no time to learn everything, but probability talks in favor of having helpful sentences in the edge of vision
  - Think more of navigating, judgement from experience than planning.

# Problem/Need
This is about the need to somehow find out what need to be done:
- Either periodically, such as at the start of the day, or after lunch.  
- Or when something else is ready, done enough, need to be paused.  

# Problem definition
This can be seen as a conversation about
- what need to be done
- and how to find out
- and who to have this conversation with
  - People such as consultants that doubles on several Design Dituations need to have this conversation with oneself
  - People that work in a Design Situation need to have it with each other
  - People from Project Management need to have it with people in the Design Situation
  
There are two parts  
- A model, visual representation that gives an overview that have the conversation around with other people.
- A way of working, an instruction, steps to follow. A way of finding out.
  - This means combing through the design workspace, catalogs, files, models on paper or as files, logs, meeting minutes. Conversation with oneself or whithin team.
  - Looking out for what need to be done. Briefly reflecting on outcome and impact. Estimating value and cost of doing. Conversation with oneself or within team.
  - What is then to be done. Judgement. Conversation with Project manager or Stakeholder, both outside the Design Situation.

# Other things
- There is a need to have a collection of problems to be solved. These are not only problems, but also needs, ideas, necessities/todo's. The important is to get an overview, to clearly describe, define so that it is possible to reach the next purpose or outcome, which is to be able to discuss how the need, problem, idea shall be taken further. This needs to be discussed on several levels, overview, understand some aspects, go deeper, start working on a solution etc. E.g. if a manual of some kind shall be written, it is tempting to just start writing. Instead, a conversation must be started around what the purpose of this manual is. Who the readers are, if they are homogenous or of different kind. Then what the manual shall be about and before 'what it shall contain', start with 'what shall it tell'. Then a discussion about form is needed, textual, a webpage, an application - there are so many different possibilities.  
- What shall be done, value etc. must be described on a purpose level, why it is needed. Not how. But, anyhow, this is not how, but why and purpose.   
There are other useful formats such as 'User Story Mapping'(Patton)  
- It shall be kept separate, but with dependency towards the project planning, which might be a backlog, a project plan etc. But, since this is closely related to how work is done, and since design work is systemic in nature, it must be possible to quickly traverse from one problem, or level of the problems, to another, quickly, a normal backlog or project plan is to rigid. Relation towards Project Management can be done via their backlog, Kanban preferrably. This is often a bit too narrow, something else, some other kind of tool is normally needed, such as a graphical view, Miro or Visio/draw.io or similar which can be handled by the Design Situation.  
- All these things need a place to have the conversation, a conversation that might take a lot of time, that is interrupted by other work, that involves people with different availability. 
- There are of course other forms, e.g. Torres OST.

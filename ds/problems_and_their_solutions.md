# Problems and their solution in the realm of digital products design
The very basic workflow is that there is a problem and we find a solution for it.  
But, what is the problem? Do we understand it enough to find a solution. Do we solve the problem right? Do we solve the right problem? E.g. is a model or some other kind of work we do, understandable, and for all people in the team, for the stakeholders or sponsors. Often we just do some kind of deliverable and dont reflect, or validate that it also works. This is also a part of the problem definition. 
Then what is the solution. We normally dont really understand what the problem really was, until we start using the solution. And is the solution really working, technically in all aspects. Is it useful. And is it commercially viable.  
Somewhere along all these lines we often get lost. Even with rather small projects we seem to get lost. So how do we actually design digital products that work.  
- We need a way to start a conversation, with users, builders, stakeholders.
- We then need a way to organise the design work, and it need to be done differently, independent and moving freely but in phase with project management work.
- We need to have a learning, discovery way of working.

We need to think the other way around. Always try to build if possible. If not, start with understanding the problem but you never will until you build something. This is beacuse there are two completely different ways of learning, one is to analyse, the other is to experience. Start building a solution, because you will eventually anyway. You will almost instantly experience different things, and then you can start understanding the problem together with the analysing part.

# Defining problems
Problems must be identified, clearly present through descriptions, a Post-It, a model or sketch.
- Problems must be defined. 
- A problem definition is very seldom, as is often the case, a one-liner, instead it needs room and space, in text, models or other ways, often a combination.
- A problem must easily be identified, e.g. via ID's, prefix and a running number, or a name, so that they can be easily referred to when planning or during discussions.

# How it is done
A good start is the book 'Conceptual Blockbusting'.   
A good way of working is to start with an overview. Then this can be further worked on, either as peeling an onion, in iterations and incremental depth. Or by spikes or T-Shapes, working on a particular aspect gaining understanding which normally gives also understanding on adjacent parts of the problem.  
The Edgy and EDF frameworks have 'Enterprise Scan' which is a good method, taking the entire architectural framework, all elements and aspects and do a shallow, flyby of the problem. This can then be further worked on, understood.

# A method, the first try
- Establish all the necessary for a 'design situation', aka 'design thing'(A.Telier)
  - Some kind of physical place. Some kind of digital surface. A text editor, Word, .markdown, graphical such as Visio, draw.io
  - Establish team, start up conversation with Project manager, Sponsor, other Team members. Rough sketch of meeting, form of those including agenda and MoM's, at least a daily standup and retros.
<_>
- Do a first 'Outline'(Kieslowski Springboard) of the problem/assignment at hand
  - For the Problem space, use design.log and start describing that a first problem statement/pitch as first entry in problem space
  - Do a first 'enterprise scan', using Edgy(other tools can be used and is tbd, such as TOGAF/Archimate). Edgy is weak on business so add those cost/income according to Coplien
    - The full Edgy elements+busines is used, however very shallow. Describe what need to be done in Problem space, using Conversation on Kieslowski 'Outline' level. 
      - Now we have a first conversation started. Spend some hours on this very first 'fill in the blanks' using Edgy Enterprise Scan+Business.
<_>
- Do also a first setup of the Learning Loop(Cool method way)
  - Identify and write down as Problem in problem space, that 'avbildning' in this state is to be done using Edgy Enterpreise Scan+Business. Capture a first typical set of People, PM and Sponsor is given, find typical domain experts, end users, write down as Problem space/or as a Log. 
  - From the very first 'Avbildning', 'build' a very first, rough 'the Real' using ABCDesEnt-pattern as 'Application Structure' since this is a 'smidig' way to easily build 'the Real'
  -  A first workshop, present 'Avbildning' and 'the Real'. Preferrably in increments, first PM, Sponsor and Team. Then test on Users. Make sure a conversation is established, via mail, chat, agenda to workshop/MoM's, go through Problem space and that this is the first take at Solution. Then show 'the Real' in its first raw cut. Learn by writing down in Log and revisit through meetings, within Team, reflect by yourself as ArkDes-role.
  - A first learning is done. Note that putting too much effort in establishing 'Hypothesis' and 'Experiment' really doesnt work, dont be too formal on this. Instead, rely on writning down in Problem space and Log and continously revisit, let important things rise to the surface, analyse then, do not try to structure or organize. Rely on conversation to be open, transparent and clear. Use 'Judgement' to guide in work. Learning is hard, revisit need to be done continously - then learning will happen.
  - Besides Edgy/Enterpreise Scan and ABCDesEnt as pattern, use other frameworks/methods/ontologies to frame, to put a grid over the problem space, for guidance and some order.
  <_>
  So what have we got now:
  - We have established the Conversation. We have described 'the Assignment'/'Pitch' in form of a first 'Avbilidning' and 'the Real'. We have described learning in Log and we have a first set of entries in Problem Space
  - We can now revisit, continously design.log and Problem Space and Logs together with PM so that project management can be kept in a first state of sync but the design process move freely. The conversation can continue, always clear on what part in Problem space we are working at. Continously revisit Log and the expanding Problem space.
  - People, Team and Users are involved
  <_>
  Then:
  - Continue with 'Treatment'-level. Repeat above in one of two ways:
    - The T-shape. Identify and re-frame moving through different aspects from Edgy or other Ontologies. Rely on that T-Shape will give fallout, more shallow, but broad. 
    - Peeling the Onion. Move in levels and increments, shallow, down through the Edgy 'Avbildning'. Rely on that all aspects will be managable if only shallow peels enough.
  - Continously expand Problem Space. Be clear in conversation what part we are working at. Do the learning loop via People, continously build on 'the Real' in form of Models, Mockups, Prototypes.
  - Keep conversation, with PM about keeping project in sync. Use judgement and learning on which problem parts to focus on and which to skip. Use Edgy+Business 'avbildning' that is emerging. 
  - Continoue on 'Treatment level' until 'seeds of action and dialogue' appears and transfer these as more  formal requirement to feed into 'build' of the real product.
<_>
Additionals:
- Use the Indigo model when doing 'the Learning Loop'. See 'the Real' as both 'Design', 'How to build' and 'Product' parts and work on the parts most needed.
- Use the idea of 'the Gravity Well' i.e. start on 'the Real' until stuck, and need to revisit Problem Space.
